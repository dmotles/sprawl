# `azure/` ‚Äî Phase-0 hub on Azure Container Apps

The first concrete Terraform root (QUM-879). Composes the four capability
modules in [`../modules/`](../modules/README.md) into a real ACA deployment of
`hubd`: a Container App behind managed HTTP/2 ingress, backed by managed
Postgres, a private blob container, and a Key Vault, all reached via a
user-assigned managed identity (AAD end-to-end ‚Äî no account keys).

See [`docs/design/hub/06-iac.md`](../../../../../docs/design/hub/06-iac.md) and
[`08-deployment.md`](../../../../../docs/design/hub/08-deployment.md).

## Public-repo / corporate-Azure rules (HARD)

- **Nothing account/employer-specific is committed.** Only `*.example`
  templates are tracked; real values live in gitignored `terraform.tfvars` /
  `backend-config.hcl`. No subscription/tenant ids, resource names, region, or
  endpoints in the tree.
- **Every resource is pinned to `var.subscription_id`** via the provider ‚Äî never
  the CLI's active default.
- **Mandatory 4-tag map** (`owner`, `long_running`, `department`, `purpose`) is
  applied to the RG **and every resource** (Azure tags do NOT inherit). The
  `tags` variable enforces the four keys are present.
- **No secret VALUE is ever a Terraform input variable or committed.** Generated
  keys are born in TF (encrypted remote state only); externally-set values are
  injected out-of-band into Key Vault.

## What it provisions

Glue (root): resource group ¬∑ Log Analytics workspace ¬∑ Container App
Environment ¬∑ Azure Container Registry (`admin_enabled = false`) ¬∑
user-assigned managed identity (UAMI) ¬∑ `AcrPull` role on the ACR for the UAMI ¬∑
`Storage Blob Data Contributor` role on the storage account for the UAMI.

Capabilities (modules): `database` (Postgres Flexible Server) ¬∑ `object-store`
(storage account + private blob container) ¬∑ `secrets` (Key Vault + grants) ¬∑
`container-host` (the `hubd` Container App, single revision / single replica,
liveness `/healthz` + readiness `/readyz`).

The UAMI + `AcrPull` are created **before** the Container App (`depends_on`) so
the first apply can pull the real image without a system-identity
chicken-and-egg.

## Secrets provisioning + the base64 alphabet foot-gun

Two generated keys use **OPPOSITE base64 alphabets** ‚Äî the most important
correctness detail on this root (both are minted in TF with the right encoding
so an operator can't fat-finger it):

- **`SPRAWL_HUB_SECRET_URL`** ‚Äî the gocloud.dev keeper URL (host-token pepper
  sealer). When `base64key://<key>` is used, the key MUST be **URL-safe** base64
  (gocloud localsecrets uses `base64.URLEncoding`): a standard-base64 key with
  `+`/`/` silently decodes to the WRONG 32 bytes (a `/` truncates the URL host),
  so minted host tokens won't verify. TF translates `+/ => -_`. If minting by
  hand: `openssl rand -base64 32 | tr '+/' '-_'`. **Must be identical between
  `hubd` and `sprawl hub token create`.** In prod, point this at a cloud
  KMS/Key Vault keeper instead of `base64key://`.
- **`SPRAWL_HUB_COOKIE_KEY`** ‚Äî OPPOSITE: **standard** base64
  (`base64.StdEncoding` in `login.go`), ‚â•32 decoded bytes. A URL-safe value
  silently disables browser login. By hand: `openssl rand -base64 32`.

`SPRAWL_HUB_DSN` comes from the Postgres module (admin password via
`random_password`), written to KV by the root. `SPRAWL_HUB_LOGIN_TOKEN` is any
non-empty shared string, set out-of-band after apply:

```bash
az keyvault secret set --vault-name <kv> --name hub-login-token \
  --value "<a-non-empty-shared-string>" --subscription <sub>
```

The `secrets` module never accepts secret VALUES ‚Äî only names. Generated values
are recorded (unavoidably) in the **encrypted remote state**; that is why the
QUM-870 state backend is remote + AAD-access-controlled.

## Apply runbook (weave/user-gated ‚Äî do NOT apply autonomously)

```bash
cd deploy/hub/infra/terraform/azure
cp terraform.tfvars.example terraform.tfvars        # fill in real values (gitignored)
cp backend-config.hcl.example backend-config.hcl    # QUM-870 backend values (gitignored)

terraform init  -backend-config=backend-config.hcl
terraform plan  -var-file=terraform.tfvars          # review the resource inventory

# Build + push the hubd image so the app can pull it (UAMI has AcrPull):
az acr build --registry <acr> --image hubd:<tag> --subscription <sub> \
  --file cmd/hubd/Dockerfile .

# Present the inventory + these exact --subscription-pinned commands for review,
# THEN:
terraform apply -var-file=terraform.tfvars -var container_image=<acr>.azurecr.io/hubd:<tag>

# Post-apply: set the out-of-band login token, then exercise the Phase-0 gate
# (sprawl enter --hub-url <hub_url> with a real token ‚Üí ListInstances / /debug/state).
```

Every `az`/`terraform` invocation pins `--subscription` / `subscription_id`.
