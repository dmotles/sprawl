## Summary

Research on how to enable continuous development on main while choosing when to cut official numbered releases.

## Recommendation: GoReleaser v2 + GitHub Actions on Tag Push

The universal standard for Go CLI tools. Push to main freely, tag when ready to release.

### Workflow
1. Develop on main continuously
2. When ready to release: `git tag v0.1.0 && git push --tags`
3. GitHub Actions detects the semver tag
4. GoReleaser builds multi-platform binaries
5. GitHub Release created automatically with changelog

### Version Strategy
- Start at `v0.1.0` (pre-1.0 signals "not yet stable")
- Semantic versioning: major.minor.patch
- Version embedded in binary via `ldflags` at build time

### Platforms
- Linux amd64, Linux arm64
- macOS amd64, macOS arm64 (Apple Silicon)
- No Windows (tmux dependency)

### Bonus
- `go install github.com/<org>/<name>@latest` works automatically once tags exist

### Config Files Needed
- `.goreleaser.yaml` — GoReleaser configuration
- `.github/workflows/release.yml` — GitHub Actions workflow
- Version embedding code in `main.go` via `ldflags`
- Makefile additions for local version injection

### Changelog
- Auto-generated from commit messages between tags
- Conventional commits recommended but not required

## Reference Projects
- gh CLI (cli/cli) — GoReleaser + GitHub Actions

- Both follow this exact pattern
